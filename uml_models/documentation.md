# UML Models Documentation

## 1. Alert Generation System
The resolved flag in Alert allows the system to keep track of whether an alert has been acknowledged. This is important in a real hospital setting where ignored alerts can be dangerous.The Alert Generation System is based on a clear separation of the data evaluation, alert creation, and alert dispatching. The AlertGenerator is the main component — it continuously compares incoming patient data to the thresholds defined in AlertThreshold objects. Each threshold is associated with a particular patient and metric type, allowing for customised alert rules rather than a one-size-fits-all limit. When the threshold is exceeded, the AlertGenerator generates an Alert object, containing the ID of the patient, the condition that triggered the alert and a timestamp. This alert is then sent to the AlertManager, which is responsible for routing and tracking all active alerts. This allows the generator to focus solely on evaluation logic, as the AlertManager is the single point of contact for the medical staff. This separation is consistent with the single responsibility principle and makes it easy to add new alert types without touching the dispatch logic.

## 2. Data Storage System
The core DataStorage class is at the heart of the Data Storage System and serves as the main repository for all patient records. Each record is a PatientRecord object with value, type and timestamp, keeping individual data points small and focused. DataStorage makes retrieval efficient by using an internal map structure that groups records by patient id. The system can automatically delete old records using deleteOldRecords, thanks to a retentionDays field, which is important both for storage management and patient privacy compliance. The DataRetriever class is there to decouple reading operations from the internals of storage. Medical staff query data through DataRetriever and not DataStorage. This helps avoid abuse of the storage layer and makes it easier to change the way data is stored without affecting the query logic. The AccessController provides a simple security level by checking if a specific role is allowed to read, write, or delete records. This is important in a hospital environment since not all staff should have the same access rights.

## 3. Patient Identification System
The Patient Identification System guarantees the correct matching of each data coming from the simulator with a real hospital patient. The PatientIdentifier is the entry point – it takes a simulator ID, and has to return the correct HospitalPatient. Instead of performing this lookup itself, it passes it to the IdentityManager which holds the complete mapping of simulator IDs to the hospital records. This delegation keeps the PatientIdentifier simple and the mapping logic easy to test in isolation; The HospitalPatient class contains sensitive information like name, medical history etc. with the private visibility. Thus it is only accessible in a controlled manner. The IdentityManager also accounts for edge cases: If a simulator ID does not match with any hospital record, then handleMismatch is called and the anomaly is logged. This is important in a safety critical system where unmatched data should never go unnoticed. The design is easily extensible, e.g., to add a database-backed implementation of IdentityManager without changing PatientIdentifier at all.

## 4. Data Access Layer
The Data Access Layer is the interface between the external simulator and the rest of the CHMS. The primary design decision is the DataListener interface, which defines a common contract to be followed by all input sources. This means that the rest of the system never needs to know whether data is coming from TCP, WebSocket, or a file, it’s always going through the same interface. The concrete listeners take care of the details of their protocol, such as handling a socket connection or watching a directory for new files. When raw data is received it is passed to DataSourceAdapter which orchestrates the next steps. The adapter invokes DataParser to convert the raw string into a structured PatientRecord. The format is automatically detected as JSON or CSV. The parsed record is then forwarded to DataStorage. This design makes the system easy to extend — adding a new data source only requires implementing DataListener without touching any other component. It also isolates parsing logic in one place, making format changes straightforward.
